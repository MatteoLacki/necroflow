# Necroflow vs other workflow systems

Comparison emphasises developer ergonomics. Necroflow is designed as a Python library for local, exploratory bioinformatics workflow development rather than as a cluster/cloud platform.

| Axis | Necroflow | Snakemake | Nextflow | Luigi | CWL/WDL | Prefect/Airflow |
|---|---|---|---|---|---|---|
| **Primary niche** | Small, Python-native, typed, content-addressed pipelines that run locally and remain inspectable as ordinary Python objects. | Mature, file-pattern-based bioinformatics workflows with strong ecosystem support and scalable executors. | Production-scale dataflow workflows with process/channel composition, JVM runtime, and strong HPC/cloud portability. | Python batch-job orchestration with class-based tasks and a central scheduler. | Portable workflow specifications for cross-runner and cross-institution execution. | General data-engineering orchestration, observability, retries, and scheduled operations. |
| **Authoring model** | Plain Python API: register `Rules`, assign `Node`s to a `Pipeline`, add many pipelines to a `DAG`. Pipeline factories read procedurally: create a node, pass it to the next rule, branch with normal Python. | Snakefile DSL: rules declare `input`, `output`, `params`, `shell`; dependencies are inferred by matching filenames and wildcards. | Groovy-like Nextflow DSL on the JVM; processes communicate through channels and workflows compose processes. | Python subclasses of `luigi.Task`; dependencies from `requires()`, outputs from `output()`, work in `run()`. | Dedicated declarative languages: YAML/JSON CWL documents or WDL task/workflow files. | Python decorators or DAG classes; task graph often centred on functions and scheduler metadata rather than file lineage. |
| **Configuration surface** | One job TOML holds the pipeline factory, requested outputs, and all user parameters. Output paths and provenance are derived from lineage rather than extra config files. | Usually a config file plus Snakefile targets, wildcard conventions, and path templates. Large projects may also use profiles, modules, sample sheets, and rule-specific parameter files. | Configuration is powerful but often split across `nextflow.config`, profiles, params files, launch flags, and channel/sample metadata. | Parameters are task-class fields or scheduler/CLI inputs; output locations remain task code conventions. | Workflow inputs are separate from task/workflow documents; portability favours explicit input objects and runner-specific launch metadata. | Configuration is deployment- and platform-oriented; scientific file identity and output layout must be modelled by the application. |
| **Rule and value model** | `Rules` are reusable typed transformations; calling a rule creates concrete `Node` values for one pipeline context. The same rule can be reused across many pipeline factories without subclassing or copying path logic. | Rules are reusable within a Snakefile or module system, but rule reuse is coupled to file-pattern conventions, wildcards, and target naming. | Processes are reusable workflow components, but reuse requires compatible channel shapes and tuple/value contracts. | The task class combines reusable behaviour, parameter schema, dependency declaration, output identity, and execution. Reuse often means subclassing or composing task classes. | Tasks/tools are reusable declarations, but composing them repeatedly is verbose and tied to the workflow language. | Task functions are reusable, but the engine does not supply typed file-product values or lineage-derived output paths by default. |
| **Simple branch** | `if config.call: P.result = r.call(P.bam)` is normal Python. The branch changes the downstream rule fingerprint while shared upstream nodes are reused. | Branches are usually encoded by target expansion, conditional input functions, config flags, or separate rules. Powerful, but control flow is split between DSL sections and wildcard resolution. | Branches are expressed through channel operators and workflow conditionals. Scales well for fan-out patterns but requires reasoning about channel shape and process invocation order. | Branches live in `requires()` or dynamic dependencies; dynamic dependencies require idempotent `run()` methods because execution can resume from the start. | Branches require explicit workflow language constructs and typed declarations; portable but verbose during exploratory edits. | Branches are ordinary Python, but reproducible file products and bioinformatics-specific invalidation must be added by the developer. |
| **Adding a step** | Add a rule, then assign `P.sorted = r.sort(P.bam)`. The new output path is derived automatically from the rule, command, config, parent fingerprints, and declared I/O types. | Add a rule and adjust downstream filename patterns. A wrong wildcard, target name, or path convention can surface at DAG construction time rather than at the edit location. | Add a process and wire channels. The edit is explicit but often requires channel reshaping, tuples, joins, or process input/output declarations. | Add a new task class and update `requires()` in downstream tasks. Output paths and target identity remain manual conventions. | Add or modify typed task declarations and workflow calls. The explicitness aids portability but slows iterative local development. | Add a task function and dependencies. File typing, deterministic paths, and stale-output semantics are outside the default model. |
| **Long pipelines** | Long workflows remain a sequence of local assignments: `P.bam = r.align(P.fastq)`, `P.sorted = r.sort(P.bam)`. Each line names a reusable intermediate; adding a downstream branch does not require renaming upstream paths. | Long workflows often accumulate many interacting filename patterns, wildcards, rule-order assumptions, and target expansions. Compact once stable, but exploratory refactors can require coordinated edits across rules. | Long workflows scale well operationally, but the author must keep channel contents, tuple structure, and process contracts aligned across many transformations. | Long workflows become many task classes with dependencies distributed across `requires()`, `output()`, and `run()`; there is little built-in help for domain-specific file typing. | Long workflows are explicit and portable, but the amount of declaration grows quickly. Good for handoff, less good for rapid iteration. | Long task graphs are easy to express in Python, but deterministic file layout, cache invalidation, and provenance must be designed separately. |
| **Typed I/O** | `NodeType` subclasses are checked at rule-call time with subtype polymorphism; scalar config kwargs are checked with Python types. | No. Rule I/O is primarily file paths and wildcards; semantic types such as `Bam` vs `Fastq` are naming conventions. | Common DSL2 pipelines still pass paths, values, tuples, and maps rather than Python-like domain types. | No. Targets describe storage locations; there is no built-in type contract between tasks. | CWL and WDL both have formal type systems. | Python type hints and Pydantic models can help, but orchestration engines do not normally enforce bioinformatics file-type contracts at graph construction. |
| **Early wiring-error feedback** | Calling `r.quantify(fastq)` when the rule expects `SortedBam` raises immediately in Python, before execution or scheduler submission. | Many errors are caught by DAG construction, but semantic file-type mismatches depend on filenames, wildcard constraints, or custom validation. | Channel shape and missing output errors are caught by the runtime; semantic mismatches are usually convention- or schema-driven. | Missing outputs are detected through `complete()` and target checks before downstream scheduling; semantic I/O mismatches are user code. | Static declarations catch many structural errors, but iteration requires working in the workflow language and runner. | Function signatures catch some Python-level mistakes; persistent file contracts are custom. |
| **Cache identity** | Every output path is `outdir/rule/fingerprint/filename`; the fingerprint covers the rule name, command, config, parent fingerprints, and declared I/O types. | Optional between-workflow caching hashes steps, parameters, software stacks, and raw inputs for selected cacheable rules. | Task executions are stored in a task cache and work directory; `-resume` checks task hash plus presence of outputs. | No. By default, a task is complete when its output target exists. Paths are chosen manually. | Runner-dependent call caching; not necessarily a user-visible content-addressed output layout. | Cache keys are configurable, but file lineage is not the default cache identity. |
| **Staleness model** | Mtime fast path plus SHA-256 fallback on parents. If a parent reruns but bytes are unchanged, children remain valid. Compromised `running`/`failed`/`interrupted` states force rerun. | Mtime and metadata/rerun triggers for code, input, params, and software environment; summary and list-change commands expose planned reruns. | Task hash includes task name, container/environment, inputs, script, globals, and other metadata; default file cache mode considers path, mtime, and size. | Binary by default: target exists means complete; custom `complete()` methods can implement richer logic. | Runner-dependent; often call-cache or output-existence based. | Task-state and cache-key based; stale external files require explicit modelling. |
| **Provenance near outputs** | Each output directory has `.rip/dependencies.toml`, per-output content hashes, `job.log`, `state`, and an ancestor graph. | Snakemake tracks metadata, reports, summaries, and archives; reports combine results with parameters, code, and software. | Nextflow stores task metadata in `.nextflow/cache`, work dirs, logs, trace/report files, and platform metadata; provenance is not a per-output TOML sidecar. | No. Central scheduler state and task history exist, but no default per-output provenance bundle. | Standards can express provenance, especially with RO-Crate profiles, but runner support varies. | Strong run metadata in platform UI/database; usually separate from result files. |
| **Multi-sample reuse** | Add one `Pipeline` per sample to a shared `DAG`; identical upstream nodes collapse by key automatically. | Wildcards and target expansion naturally handle large sample sets; shared products can be modelled explicitly or through cached rules. | Channels are well suited to fan-out/fan-in and streaming many samples. | Many task instances can be generated from parameters; reuse depends on manually stable task IDs and output targets. | Scatter/gather constructs support batch execution, with verbosity trade-offs. | Dynamic mapping exists, but domain-specific file reuse is manual. |
| **Inspection from Python** | `Pipeline` and `DAG` are importable objects; tests can call the factory, inspect `P.nodes`, resolve commands, and assert node types without spawning a runner. | Snakemake has APIs and dry-run/DAG commands, but the primary graph source is a Snakefile DSL. | No. Nextflow graphs are scripts executed by the Nextflow runtime; Python unit tests cannot directly inspect pipeline objects. | Tasks are Python classes and can be unit tested, though the graph is distributed across class methods. | No. Specs can be parsed, but not as native Python pipeline objects. | Python-native, although scheduler/runtime state is often external. |
| **Execution backends** | Local `ThreadPoolExecutor` with resource caps, pluggable scheduler, `keep_going`, dry-run, and autoclean. No native HPC/cloud executor yet. | Local, server, cluster, grid, and cloud execution without changing the workflow definition; executor plugin ecosystem. | Local, HPC schedulers, Kubernetes, AWS Batch, Azure Batch, and other cloud executors through configuration. | Local workers and central scheduler; Hadoop integrations; limited bioinformatics HPC/cloud support. | Depends on runner: local engines, Cromwell, Toil, Terra, REANA, and other platforms. | Strong distributed/cloud execution depending on deployment. |
| **Environment management** | Delegated to commands, host environment, or user scripts; no built-in conda/container annotations yet. | Conda, containers, wrappers, reports, and workflow archives are mature. | Containers, Conda, Spack, modules, Wave, and cloud deployment are first-class in the ecosystem. | User code or task wrappers manage environments. | Container and environment declarations are core portability features. | Infrastructure integrations exist, but scientific software environments remain user-managed. |
| **GUI / observability** | Optional local browser GUI for pipeline visualisation and target selection; job logs live next to outputs. | DAG visualisation, reports, monitoring, and community UI tooling. | Reports, traces, logs, and Seqera Platform. | Central scheduler web UI with task status, progress messages, and tracking URLs. | Runner/platform dependent. | Strong dashboards, logs, retries, and schedules. |
| **Main advantage over Necroflow** | — | Much larger ecosystem, reusable wrappers, mature production deployment, environment management, and cluster/cloud support. | Best-in-class production portability, nf-core ecosystem, channel dataflow, and cloud/HPC execution. | Very simple Python task abstraction and scheduler UI for general batch jobs. | Portability, standardisation, and formal type systems. | Operational scheduling and observability for business/data platforms. |
| **Main advantage of Necroflow** | Typed domain objects, content-addressed paths, per-output provenance, Python object inspection, and natural Python control flow in one small framework. | Avoids wildcard/path-pattern coupling for local development and catches semantic type mistakes at rule-call time. | Avoids a second JVM/Groovy language and lets Python tests inspect the graph directly. | Derives cache paths and provenance from computational lineage instead of requiring hand-written output conventions. | Trades cross-platform standardisation for a faster Python edit-test loop. | Provides bioinformatics file semantics by default rather than as application code. |
| **Best fit** | Exploratory and shared-filesystem bioinformatics projects where the author wants Python-native development, deterministic cache paths, and rich local provenance. | Published or production bioinformatics workflows requiring mature ecosystem, deployment profiles, and community practices. | Large consortia, nf-core-style pipelines, cloud/HPC portability, and complex fan-out/fan-in dataflow. | General batch pipelines where Python task classes and scheduler visibility matter more than file-lineage provenance. | Institutional portability, regulated handoff, and cross-runner interoperability. | ETL, ML operations, scheduled jobs, and service-oriented orchestration. |
